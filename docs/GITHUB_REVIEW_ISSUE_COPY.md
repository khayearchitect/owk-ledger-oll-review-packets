# GitHub Review Issue Draft - OLL Synthetic Reviewer Packet

Title option:

```text
Review request: would this synthetic business packet save a reviewer time?
```

Post body:

```text
We are testing a small idea before building anything bigger around it.

Small businesses often prepare records in fragments: invoices in one place, payment references somewhere else, client details partly updated, and missing documents only discovered after the reviewer has already started asking questions.

So we created a synthetic reviewer-ready packet to test one question:

Would this structure help a reviewer see what is present, what is missing, and what needs clarification faster than the usual back-and-forth?

The packet is intentionally synthetic. No real business, client, employee, payment, bank, tax, ID, payroll, or invoice data is included.

It includes:

- synthetic quotes and invoices
- client register data
- payment-reference and invoice-lifecycle notes
- a synthetic bank-statement summary
- a simple expense summary
- a small HR / payslip review stub
- CSV exports
- a missing-information checklist

We are not asking for tax, legal, accounting, payroll, audit, or compliance advice.

We are asking reviewers to critique the packet format itself.

Useful feedback would be things like:

1. What would still block you from starting useful review?
2. Which section made the record set easier to understand?
3. Which section felt unnecessary or distracting?
4. Which fields are missing, weak, or badly named?
5. Does the missing-information checklist save time, or does it add noise?
6. Would you prefer this as a static packet, CSV files, a link-based view, or all three?
7. Does anything in the wording sound like it overclaims professional authority?

Boundary:
Prepared by the business. Not professionally reviewed unless marked by a verified reviewer.

Repo:
https://github.com/khayearchitect/owk-ledger-oll-review-packets
```
